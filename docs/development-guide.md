# 40kCalculator 개발 지침 및 진행 현황

- 문서 상태: v0.1
- 기준일: 2026-07-02
- 대상 저장소: `chojh1027/40kCalculator`
- 관련 문서: [프로젝트 프로포절](./proposal.md), [기술 설계서](./technical-design.md)

이 문서는 프로젝트를 계속 개발할 때 지켜야 할 공통 원칙과 현재 구현 상태를 한곳에서 확인하기 위한 작업 기준서다. 기능을 추가하거나 구조를 변경할 때는 이 문서, 기술 설계서, 테스트 코드를 함께 갱신한다.

---

## 1. 프로젝트 개발 목표

40kCalculator는 워해머 40,000의 전투 결과를 단일 무작위 시뮬레이션이 아니라 **실제 이산 확률분포**로 계산하는 정적 웹 애플리케이션이다.

핵심 목표는 다음과 같다.

- 사용자가 공격 유닛, 무장, 방어 유닛을 선택해 전투 결과를 계산할 수 있게 한다.
- 평균 피해만이 아니라 모델 파괴 수별 확률과 전멸 확률을 제공한다.
- 동일한 입력과 데이터 버전에는 항상 동일한 결과를 반환한다.
- UI, 계산 엔진, 게임 데이터를 분리해 규칙과 데이터 변경에 대응한다.
- 별도 서버 없이 정적 웹사이트로 배포하고 모바일에서도 사용할 수 있게 한다.

구현 방향이 충돌할 경우 다음 순서로 판단한다.

2. 자동 테스트와 검증된 골든 테스트 결과
3. `docs/technical-design.md`의 기술 설계

규칙 해석이 불명확하면 임의로 구현하지 않고, 입력 조건과 가정을 문서 및 테스트 이름에 명시한다.

## 3. 필수 개발 원칙

### 3.1 계산 정확성을 UI 편의보다 우선한다

- 지원하지 않는 규칙을 지원하는 것처럼 표시하지 않는다.
- 근사값을 사용할 경우 근사 방법과 허용 오차를 명시한다.
- 결과 확률의 총합은 허용 오차 내에서 반드시 1이어야 한다.
- 평균값, 최빈값, 중앙값, 단일 전투 예시를 서로 혼동하지 않는다.
- 정규분포를 가정하지 않고 실제 이산 확률분포를 기본 결과로 사용한다.

### 3.2 계산 엔진은 UI와 독립적으로 유지한다

- `packages/calculator`는 React, DOM, 브라우저 저장소에 직접 의존하지 않는다.
- 계산 함수는 입력을 직접 변경하지 않는 순수 함수 형태를 우선한다.
- 같은 입력에는 항상 같은 결과를 반환해야 한다.
- 화면 표시용 문구와 번역은 계산 엔진 내부에 넣지 않는다.

### 3.3 게임 데이터와 계산 코드를 분리한다

- 유닛과 무장의 실제 수치를 계산 함수에 직접 추가하지 않는다.
- MVP의 하드코딩 프리셋은 임시 데이터로 취급한다.
- 외부 데이터 단계부터 모든 엔티티는 표시 이름과 분리된 고유 ID를 사용한다.
- 데이터에는 출처, 규칙 버전, 적용 시작일, 수정일을 기록한다.
- 배포된 데이터 릴리스는 덮어쓰지 않고 새 릴리스로 추가한다.

### 3.4 규칙은 단계별로 구현한다

전투 계산 순서는 기본적으로 다음 파이프라인을 따른다.

```text
입력 검증
→ 공격 횟수 결정
→ 명중
→ 상처
→ 내성
→ 피해 적용
→ 모델별 피해 할당
→ 동일 상태 확률 병합
→ 결과 요약
```

특수 규칙은 전체 계산식에 조건문을 누적하지 않고, 개입 단계와 우선순위가 명확한 효과 또는 처리기로 추가한다.

### 3.5 작은 단위로 변경한다

- 한 변경에서는 하나의 규칙, 하나의 데이터 구조, 하나의 UI 흐름을 중심으로 수정한다.
- 계산 엔진 변경과 대규모 UI 변경을 가능한 한 같은 작업에 섞지 않는다.
- 기능 추가 시 테스트를 먼저 작성하거나, 최소한 같은 변경에 테스트를 포함한다.
- 기존 공개 동작을 변경하면 문서와 테스트에서 변경 이유를 확인할 수 있어야 한다.

## 4. 코드 작성 지침

### TypeScript

- `strict` 모드를 유지한다.
- `any` 사용을 피하고 외부 입력은 명시적으로 검증한다.
- 단위가 다른 수치는 이름으로 구분한다. 예: `probability`, `wounds`, `modelCount`.
- 확률은 기본적으로 `0` 이상 `1` 이하의 `number`로 표현한다.
- 부동소수점 비교에는 허용 오차를 사용한다.
- 공개 함수와 핵심 상태 타입은 역할을 드러내는 이름을 사용한다.

### React

- React 컴포넌트는 입력, 상태 표시, 사용자 상호작용을 담당한다.
- 확률 계산 공식을 컴포넌트 안에 중복 구현하지 않는다.
- 계산 결과를 다시 가공할 때 의미가 바뀌지 않도록 결과 타입을 우선 사용한다.
- 모바일 화면을 기본 검토 대상으로 포함한다.
- 색상만으로 확률이나 상태를 구분하지 않는다.

### 데이터

- 표시 이름을 참조 키로 사용하지 않는다.
- ID 변경은 데이터 마이그레이션이 필요한 호환성 변경으로 취급한다.
- 중복 ID, 누락 참조, 잘못된 수치 범위는 빌드 전에 차단한다.
- 공식 설명문이나 이미지를 그대로 복제하지 않고 계산에 필요한 구조화된 정보와 자체 설명을 사용한다.

## 5. 테스트 지침

모든 계산 엔진 변경은 최소한 다음 조건을 검증해야 한다.

- 전체 결과 확률의 합이 1이다.
- 모든 개별 확률이 0 이상 1 이하이다.
- 동일 입력의 결과가 결정적이다.
- 공격 횟수가 0이면 피해와 전멸 확률도 0이다.
- 일반 피해의 초과 피해가 다음 모델로 전달되지 않는다.
- 방어 능력이 개선될 때 평균 피해가 증가하지 않는다.
- 방어 모델 수가 증가할 때 전멸 확률이 증가하지 않는다.

### 검사 명령

```bash
npm test
npm run typecheck
npm run build
npm run check
```

`npm run check`가 타입 검사, 단위 테스트, 프로덕션 빌드를 모두 통과해야 배포 가능한 상태로 판단한다.

### 테스트 종류

- **단위 테스트:** 확률 공식, 상처 기준, 내성, 피해 할당 등 작은 규칙 검증
- **불변식 테스트:** 확률 범위, 총합, 단조성 등 항상 성립해야 하는 조건 검증
- **골든 테스트:** 대표 전투 입력과 확정된 예상 결과를 파일로 보관해 회귀 검증
- **교차 검증:** 수작업 계산 또는 제한적인 내부 반복 시뮬레이션과 비교

무작위 시뮬레이션은 사용자 결과 산출 방식이 아니라 계산 엔진 검증 보조 수단으로만 사용한다.

## 6. Git 및 배포 지침

### 브랜치와 변경 관리

향후 기능 개발은 가능하면 기능 브랜치와 Pull Request를 사용한다.

권장 브랜치 예시:

```text
feat/variable-damage
feat/game-data-schema
fix/save-probability
chore/update-ci
```

권장 커밋 접두어:

```text
feat: 새 기능
fix: 오류 수정
test: 테스트 추가 또는 수정
docs: 문서 변경
refactor: 외부 동작을 바꾸지 않는 구조 개선
chore: 설정, 도구, 의존성 변경
ci: 자동화 변경
```

### CI

`main` 브랜치 푸시와 Pull Request에서 다음 검사를 수행한다.

```text
의존성 설치
→ TypeScript 타입 검사
→ 계산 엔진 단위 테스트
→ 웹 프로덕션 빌드
```

### GitHub Pages

`main` 브랜치가 변경되면 GitHub Actions가 프로젝트를 검사하고 `apps/web/dist`를 GitHub Pages에 배포한다.

배포 주소:

```text
https://chojh1027.github.io/40kCalculator/
```

배포 실패 시 다음 순서로 확인한다.

1. 저장소의 `Settings → Pages`에서 Source가 `GitHub Actions`인지 확인
2. `Actions → Deploy GitHub Pages`에서 실패 단계 확인
3. `npm run check` 실패 여부 확인
4. 빌드 산출물 경로가 `apps/web/dist`인지 확인

|---|---:|---|
| 코드 외부 게임 데이터 | ⬜ | 현재 프리셋은 `App.tsx`에 임시 정의 |
| 공통 데이터 스키마 | ⬜ | Unit, Weapon, Ability, Effect 정의 필요 |
| 런타임 스키마 검증 | ⬜ | Zod 등 도입 예정 |
| 진영별 데이터 청크 | ⬜ | 최초 다운로드 크기 최적화 필요 |
| 데이터 manifest | ⬜ | 버전, 해시, 진영 파일 정보 필요 |
| IndexedDB 저장 | ⬜ | 설치된 데이터와 선택 버전 저장 필요 |
| 과거 데이터 버전 전환 | ⬜ | 불변 릴리스 구조 이후 구현 |
| 데이터 갱신 CLI | ⬜ | `validate`, `diff`, `build`, `release` 목표 |

---

## 9. 현재 알려진 제약과 기술 부채

- 무장과 방어 유닛 데이터가 웹 컴포넌트에 하드코딩되어 있다.
- 공격 횟수와 피해가 정수 고정값으로 제한되어 있다.
- 일반적인 이산 확률분포 연산을 재사용할 공통 PMF 모듈이 없다.
- 규칙 효과의 적용 시점, 우선순위, 중첩을 관리하는 Modifier 시스템이 없다.
- 골든 테스트와 속성 기반 테스트가 아직 없다.
- 저장소에 재현 가능한 의존성 설치를 위한 lockfile 관리 정책을 확정해야 한다.
- 계산 결과를 실제 공식 예시와 대규모로 대조한 검증 자료가 아직 없다.
- 게임 데이터의 출처와 배포 가능 범위에 대한 별도 검토가 필요하다.

## 10. 다음 개발 우선순위

### 우선순위 1: 계산 코어 안정화

- 공통 PMF 자료구조와 합성 함수 작성
- 가변 주사위 표현 추가
- 골든 테스트 도입
- 확률 범위와 단조성 불변식 테스트 강화
- 의존성 lockfile과 CI 설치 방식 확정

### 우선순위 2: 수동 입력 가능한 계산기

- 공격·방어 스탯 직접 입력
- 입력 오류 표시
- 상세 피해 상태 분포 제공
- 모바일 접근성과 터치 영역 개선

### 우선순위 3: 게임 데이터 분리

- Unit, ModelProfile, WeaponProfile, Ability 스키마 정의
- 하드코딩 프리셋을 외부 샘플 JSON으로 이전
- 런타임 데이터 검증 추가
- 표시 이름과 ID 분리

### 우선순위 4: 규칙 엔진 확장

- 재굴림
- Sustained Hits
- Lethal Hits
- 가변 피해
- 피해 감소
- Feel No Pain
- 복수 공격 그룹

한 단계의 완료 조건을 충족하고 테스트가 안정된 후 다음 단계로 이동한다.

## 11. 기능 추가 체크리스트

새 규칙이나 기능을 추가할 때 다음을 확인한다.

- [ ] 지원할 공식 규칙과 적용 조건을 문장으로 정의했다.
- [ ] 계산 파이프라인에서 개입할 단계를 정했다.
- [ ] 기존 입력·출력 타입에 미치는 영향을 검토했다.
- [ ] 정상 사례와 경계 사례 테스트를 추가했다.
- [ ] 전체 확률 합과 결정성을 검증했다.
- [ ] UI에서 지원 범위와 제한을 정확히 표시했다.
- [ ] 기술 설계서 또는 이 문서를 갱신했다.
- [ ] `npm run check`를 통과했다.
- [ ] 모바일 배포 화면을 확인했다.

## 12. 이 문서의 갱신 규칙

다음 상황에서는 이 문서를 반드시 갱신한다.

- 새로운 계산 규칙을 지원했을 때
- 저장소 구조나 기술 스택을 변경했을 때
- 데이터 관리 방식이나 배포 방식을 변경했을 때
- 주요 기능의 상태가 완료 또는 부분 구현으로 바뀌었을 때
- 새로운 기술 부채나 중요한 제약이 확인되었을 때
- 다음 개발 우선순위가 변경되었을 때

진행 현황 표는 실제 코드와 테스트를 기준으로 작성하며, 계획만 존재하는 기능을 완료로 표시하지 않는다.
