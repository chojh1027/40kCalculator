## 1. 프로젝트 개발 목표

Dice Servitor는 워해머 40,000의 전투 결과를 단일 무작위 시뮬레이션이 아니라 실제 이산 확률분포로 계산하는 정적 웹 애플리케이션이다.

- 공격 진영, 팩션, 유닛, 무장과 방어 유닛을 선택해 계산한다.
- 평균 피해, 평균 파괴 모델 수, 가장 가능성이 높은 결과와 전멸 확률을 제공한다.
- 같은 입력과 데이터 버전에는 같은 결과를 반환한다.
- UI, 계산 엔진과 게임 데이터를 분리한다.
- 별도 서버 없이 정적 웹사이트로 배포한다.

사용자에게 표시하는 서비스 이름은 `Dice Servitor`이며 저장소와 내부 패키지 이름은 현재 `40kCalculator` 및 `@40k-calculator/*`를 유지한다.

## 3. 필수 개발 원칙

### 계산 정확성

- 지원하지 않는 규칙을 지원하는 것처럼 표시하지 않는다.
- 결과 확률의 총합은 허용 오차 내에서 1이어야 한다.
- 평균값, 최빈값과 단일 전투 예시를 구분한다.
- 정규분포를 가정하지 않고 실제 이산 확률분포를 기본 결과로 사용한다.
- 근사 또는 확률 가지치기를 사용하면 허용 오차를 문서화한다.

### 계산 엔진 독립성

- `packages/calculator`는 React, DOM과 브라우저 저장소에 의존하지 않는다.
- 계산 함수는 입력을 직접 변경하지 않는 순수 함수 형태를 우선한다.
- 화면 문구와 번역은 계산 엔진에 넣지 않는다.

### PMF 사용 원칙

- 새로운 가변 주사위 또는 확률 상태 합성은 `packages/calculator/src/pmf.ts`를 우선 사용한다.
- 동일한 상태는 반드시 하나의 키로 병합한다.
- 객체 상태는 `PmfKeySelector`를 명시한다.
- 음수, 무한대, `NaN` 확률과 전체 확률 질량이 0인 분포를 허용하지 않는다.
- 공개 분포는 정규화된 상태를 유지한다.

### 주사위 표현 원칙

- 고정값과 가변 주사위는 `DiceExpression`으로 표현한다.
- `fixed`는 0 이상의 안전한 정수만 허용한다.
- `dice`는 `count`, `sides`, 선택적 `modifier`를 사용한다.
- 주사위 개수와 면 수는 각각 1~100, 2~100 범위로 제한한다.
- 음수 최종 결과가 가능한 표현은 허용하지 않는다.
- 공격 횟수는 0~200, 개별 피해 결과는 1~30 범위를 적용한다.
- 여러 모델이 같은 가변 공격 무장을 사용할 때 각 모델의 공격 주사위는 독립적으로 합산한다.
- 피해 주사위는 모델 수로 반복하지 않고 실패한 내성마다 독립적으로 적용한다.

### 피해 할당 원칙

- 각 실패한 내성은 하나의 피해 이벤트다.
- 일반 피해의 초과 피해는 다음 모델로 전달하지 않는다.
- 이미 피해를 입은 모델이 있으면 다음 피해 이벤트를 그 모델에 적용한다.
- 모델이 파괴되면 다음 피해 이벤트는 새 모델의 전체 운드에서 시작한다.
- 동일한 파괴 모델 수·현재 모델 잔여 운드·전멸 상태는 하나의 PMF 상태로 병합한다.

### 데이터 분리

- 유닛과 무장 수치를 계산 함수에 직접 추가하지 않는다.
- 현재 `apps/web/src/data/catalog.ts`는 임시 샘플 카탈로그다.
- 표시 이름을 참조 키로 사용하지 않는다.
- 유닛은 무장 ID를 참조하고 무장 타입은 무장 데이터에 둔다.
- 정식 외부 데이터에는 출처, 규칙 버전, 적용일과 수정일을 기록한다.
- 명칭에 `(Temporary)`가 포함된 샘플은 계산 검증용이며 정식 게임 데이터로 취급하지 않는다.

### 작은 단위의 변경

- 한 변경에서는 하나의 규칙, 자료구조 또는 UI 흐름을 중심으로 수정한다.
- 계산 엔진 변경에는 같은 변경에서 테스트를 추가한다.
- 공개 동작을 변경하면 테스트와 문서에서 이유를 확인할 수 있어야 한다.

## 4. 코드 작성 지침

### TypeScript

- `strict` 모드를 유지한다.
- `any` 사용을 피하고 외부 입력을 명시적으로 검증한다.
- 확률은 `0` 이상 `1` 이하의 `number`로 표현한다.
- 부동소수점 비교에는 허용 오차를 사용한다.
- 공개 타입과 함수 이름은 역할을 명확히 드러내야 한다.
- 내부 상태 배열은 읽기 전용으로 노출한다.
- 판별 유니온의 `kind` 값은 안정적인 문자열 리터럴을 사용한다.

### React

- React 컴포넌트는 입력, 상태 표시와 상호작용을 담당한다.
- 확률 계산 공식을 컴포넌트 안에 중복 구현하지 않는다.
- 상위 선택값이 변경되면 하위 선택값을 유효한 값으로 갱신한다.
- 모바일 화면과 키보드 접근성을 함께 검토한다.
- 색상만으로 상태를 구분하지 않는다.

### 데이터

- ID 변경은 호환성 변경으로 취급한다.
- 중복 ID, 누락 참조와 잘못된 수치 범위는 빌드 또는 로딩 전에 차단한다.
- 공식 이미지와 긴 설명문을 복제하지 않는다.

## 5. 테스트 지침

모든 계산 엔진 변경은 가능한 범위에서 다음을 검증한다.

- 모든 개별 확률이 유한한 0 이상 값이다.
- 전체 확률 합이 1이다.
- 동일 입력의 결과가 결정적이다.
- 동일 상태가 중복 출력되지 않는다.
- 고정 주사위 표현은 확률 1의 단일 결과를 반환한다.
- D3와 D6는 균등분포를 반환한다.
- 복수 주사위의 합은 정확한 조합 확률을 반환한다.
- 주사위 보정치가 결과 범위와 기대값에 반영된다.
- 모델 수만큼 가변 공격 표현을 반복하면 주사위 개수와 보정치가 함께 반복된다.
- 고정 숫자와 고정 `DiceExpression`의 공격·피해 결과는 동일하다.
- D6 피해는 실패한 내성마다 독립적으로 적용된다.
- 일반 피해의 초과 피해가 다음 모델로 전달되지 않는다.
- 공격 횟수가 0이면 최종 피해와 전멸 확률도 0이다.
- 방어 능력이 좋아질 때 평균 피해가 증가하지 않는다.
- 방어 모델 수가 늘어날 때 전멸 확률이 증가하지 않는다.

### 테스트 계층

```text
pmf.test.ts              PMF 자료구조 단위 테스트
dice-expression.test.ts  주사위 표현 단위 테스트
index.test.ts            전투 규칙 단위·회귀 테스트
golden.test.ts           대표 전투 결과 고정\ invariants.test.ts        확률·상태·결정성·단조성 검증
```

### 골든 테스트 기준

대표 조합은 계산 결과의 다음 값을 고정한다.

- 평균 공격·명중·상처·실패 내성
- 실패 내성당 평균 피해
- 평균 유효 피해
- 평균 파괴 모델 수
- 전멸 확률
- 가장 가능성이 높은 방어 상태와 그 확률

현재 골든 조합:

- 고정 공격 + 고정 피해
- D6 공격 + 고정 피해
- 고정 공격 + D3 피해
- 고정 공격 + D6 피해
- D6 공격 + D6 피해

골든 값 변경은 규칙 변경 또는 버그 수정 근거가 있을 때만 허용한다.

### 불변식 테스트 기준

- 모든 공개 PMF의 확률 합은 허용 오차 내에서 1이다.
- 값 분포와 방어 상태 분포에 중복 키가 없다.
- 전멸 상태는 파괴 모델 수가 목표 모델 수이고 잔여 운드가 0이다.
- 비전멸 상태의 잔여 운드는 1 이상 최대 운드 이하다.
- `destroyedModelDistribution.atLeastProbability`는 단조 감소한다.
- 분포에서 다시 계산한 기대값은 요약 기대값과 일치한다.
- 내성이 좋아지면 평균 피해와 전멸 확률이 증가하지 않는다.
- 모델 수가 증가하면 전멸 확률이 증가하지 않는다.
- 일반 피해 이벤트 한 번은 모델 하나만 파괴할 수 있다.

## 8. 현재 알려진 제약과 기술 부채

- 단일 공격 그룹만 처리한다.
- 재굴림, 치명타, 추가 명중과 별도 치명상 경로가 없다.
- 피해 감소, 최소 피해와 Feel No Pain이 없다.
- 임시 D6 무장은 정식 게임 데이터가 아니다.
- 유닛과 모델 프로필이 하나의 `Unit` 타입에 결합돼 있다.
- 게임 데이터가 애플리케이션 번들에 포함된다.
- 속성 기반 테스트 프레임워크는 아직 없다.
- lockfile 정책이 확정되지 않았다.
- 실제 게임 데이터 출처와 배포 범위 검토가 필요하다.
- `Models using this weapon`은 모델당 동일한 무장 하나를 가정한다.
