# Dice Servitor 개발 로드맵

- 문서 상태: v0.1
- 기준일: 2026-07-03
- 대상 저장소: `chojh1027/40kCalculator`
- 관련 문서: [프로젝트 프로포절](./proposal.md), [기술 설계서](./technical-design.md), [개발 지침 및 진행 현황](./development-guide.md)

이 문서는 Dice Servitor의 현재 구현 기준선과 앞으로 진행할 개발 순서를 관리한다. 장기적인 제품 목적은 프로포절에서, 구조와 기술적 결정은 기술 설계서에서, 개발 원칙과 세부 구현 상태는 개발 지침에서 관리한다.

## 2. 현재 단계 평가

현재 프로젝트는 **기본 전투 확률 계산 MVP와 선택형 웹 UI가 동작하는 단계**다.

계산 엔진은 고정 공격 횟수와 고정 피해를 사용하는 단일 공격 그룹에 대해 명중, 상처, 내성, 피해 할당을 실제 이산 확률분포로 계산한다. 다중 운드 모델과 일반 피해의 초과 피해 비전달(non-spill)을 처리하며, 평균 유효 피해, 평균 파괴 모델 수, 가장 가능성이 높은 결과, 전멸 확률과 단계별 분포를 반환한다.

웹 UI는 영어로 제공되며, Alliance → Faction → Unit → Weapon 순서로 공격 대상을 구성하고 방어 유닛을 선택할 수 있다. 서비스 표시 이름은 `Dice Servitor`다.

유닛과 무장 데이터는 `apps/web/src/data/catalog.ts`로 UI 컴포넌트에서 분리됐고, 유닛은 `weaponIds`로 무장을 참조한다. 무장의 원거리·근거리 구분도 무장 데이터에 존재한다. 다만 데이터는 아직 TypeScript 번들에 포함된 샘플 카탈로그이며 외부 JSON, 런타임 스키마 검증, 데이터 릴리스 및 로컬 저장은 구현되지 않았다.

### 2.1 완료된 기반

| 영역 | 상태 | 현재 기준선 |
|---|---:|---|
| 정적 웹 애플리케이션 | ✅ | React + TypeScript + Vite |
| 계산 엔진 분리 | ✅ | `packages/calculator`가 UI와 독립적으로 동작 |
| 기본 전투 계산 | ✅ | 명중, 상처, 내성, 고정 피해, 모델별 피해 할당 |
| 실제 이산 확률분포 | ✅ | 무작위 표본이 아닌 가능한 결과 확률 합산 |
| 핵심 결과 요약 | ✅ | 평균 피해, 평균 파괴 수, 최빈 결과, 전멸 확률 |
| 단위 테스트 | ✅ | 결정성, 확률 정규화, non-spill, 0회 공격 등 |
| GitHub Actions CI | ✅ | 타입 검사, 테스트, 빌드 |
| GitHub Pages | ✅ | 모바일 접근 가능한 정적 배포 |
| 영어 UI | ✅ | 기본 화면 문구 영어화 |
| 서비스 이름 | ✅ | `Dice Servitor` 적용 |
| 선택 흐름 | ✅ | Alliance, Faction, Unit, Weapon 선택 |
| 유닛·무장 분리 | 🟡 | 타입과 카탈로그는 분리됐으나 외부 데이터는 아님 |
| 원거리·근거리 구분 | ✅ | `Weapon.type`에 저장 |

### 2.2 현재 핵심 제약

- 공격 횟수와 피해가 고정 정수다.
- 단일 공격 그룹만 지원한다.
- 재굴림, 치명타, 추가 명중, 치명상, 피해 감소, Feel No Pain을 지원하지 않는다.
- 일반적인 이산 확률분포 연산을 재사용할 공통 PMF 모듈이 없다.
- 유닛과 모델 프로필이 하나의 `Unit` 타입에 결합돼 있다.
- 샘플 게임 데이터가 TypeScript 카탈로그로 빌드에 포함된다.
- 런타임 데이터 검증, 데이터 버전, IndexedDB, 갱신 도구가 없다.
- 골든 테스트와 속성 기반 테스트가 없다.
- 실제 게임 데이터의 출처와 배포 범위 검토가 남아 있다.

## 11. 상시 과제

### 계산 정확성

- 공식 규칙 해석과 적용 순서를 테스트 이름과 문서에 남긴다.
- 근사 또는 확률 가지치기를 도입하면 허용 오차를 명시한다.
- 실제 공식 예시 또는 수작업 계산과 교차 검증한다.

### 데이터 권리와 출처

- 데이터 출처와 확인 날짜를 기록한다.
- 공식 이미지와 긴 설명문을 복제하지 않는다.
- 수치 데이터와 자체 작성 설명의 배포 가능 범위를 별도로 검토한다.

### 문서 유지

- 기능 상태가 변경되면 본 문서와 `development-guide.md`를 함께 갱신한다.
- 구조 또는 기술적 결정이 변경되면 `technical-design.md`를 갱신한다.
- 제품 목표 또는 개발 범위가 변경되면 `proposal.md`를 갱신한다.
