## 1. 프로젝트 목적

워해머 40,000 미니어처 게임에서는 부대 편성, 공격 대상 선택과 전술 수립을 위해 유닛과 무장의 전투 결과를 반복해서 계산해야 한다. 공식 자료나 데이터시트에서 수치를 확인한 뒤 직접 확률을 계산하는 과정은 번거롭고, 가변 공격·가변 피해·재굴림·Critical Hit·추가 명중 같은 규칙이 결합되면 수작업 계산은 빠르게 복잡해진다.

Dice Servitor는 공격 유닛, 무장과 방어 유닛을 선택하면 가능한 전투 결과를 실제 이산 확률분포로 계산하는 도구다. 단일 무작위 시뮬레이션 결과 대신 평균값, 최빈 결과, 결과별 확률과 전멸 확률을 제공하여 사용자가 전술적 판단에 활용할 수 있도록 한다.

## 2. 문제 정의

### 반복적인 데이터 확인

기존 계산 흐름에서는 사용자가 유닛 능력치, 무장 프로필과 특수 규칙을 별도 자료에서 확인한 뒤 계산기에 다시 입력해야 한다.

### 복잡한 규칙 결합

명중, 상처, 내성, 피해와 모델별 피해 할당뿐 아니라 재굴림, Critical Hit, Sustained Hits와 Lethal Hits 같은 규칙의 적용 순서가 결과를 바꾼다.

### 확률분포의 비정규성

모델별 피해 할당과 초과 피해 소멸 때문에 결과가 정규분포를 따른다고 가정할 수 없다. 평균 피해만으로는 전멸 가능성이나 모델 파괴 결과의 편향을 설명하기 어렵다.

### 지속적인 게임 데이터 변경

게임 업데이트에 따라 유닛, 무장과 규칙 데이터가 바뀐다. 계산 코드에 수치를 직접 작성하면 데이터 갱신마다 애플리케이션 코드를 수정해야 한다.

## 3. 프로젝트 목표

- 공격 유닛, 무장과 방어 유닛 선택 중심의 계산 흐름을 제공한다.

- 평균 유효 피해, 평균 파괴 모델 수, 최빈 결과와 전멸 확률을 제공한다.
- 일반 명중, Critical Hit, 추가 명중, 상처 굴림과 자동 상처를 구분해 설명 가능한 결과를 제공한다.

- 이후 데이터 버전, 진영별 청크와 로컬 캐시를 추가할 수 있는 구조를 유지한다.

## 4. 계산 결과 산출 원칙

### 실제 이산 확률분포

가능한 결과 경로를 확률 질량 함수(PMF)로 합성한다. 주사위 결과를 임의로 한 번 생성하는 방식은 사용자 결과로 사용하지 않는다.

### 결정성

동일한 전투 입력과 동일한 데이터에는 항상 동일한 결과를 반환한다.

### 결과 설명 가능성

결과는 다음 단계를 구분한다.

```text
공격 횟수
→ 일반 명중 / Critical Hit
→ Sustained 추가 명중
→ 상처 굴림 수 / Lethal 자동 상처
→ 총 상처
→ 실패 내성
→ 피해
→ 모델 파괴와 잔여 운드
```

### non-spill 피해

일반 피해는 한 모델의 남은 운드를 초과하더라도 다음 모델로 전달되지 않는다. 다중 운드 모델의 상태는 파괴된 모델 수와 현재 모델의 남은 운드로 추적한다.

### 평균과 대표 결과의 구분

`모델 3개 파괴 + 다음 모델 1W 남음` 같은 정수 상태는 평균이 아니라 가장 가능성이 높은 결과다. UI에서는 평균값과 최빈 결과를 명확히 구분한다.

### 정규분포 비가정

결과를 정규분포로 근사하지 않고 실제 이산분포를 막대그래프로 표시한다.

## 6. 현재 데이터의 성격

현재 카탈로그는 구조와 계산 경로를 검증하기 위한 자체 작성 샘플 데이터다. 공식 전체 게임 데이터가 아니며, 임시 테스트 무장이 포함되어 있다.

```text
3 Alliances
4 Factions
7 Model Profiles
7 Units
11 Weapon Profiles
2 calculable Abilities
```

공식 자료의 이미지나 긴 설명문을 복제하지 않고 계산에 필요한 구조화된 수치와 자체 작성 설명을 중심으로 관리한다. 실제 게임 데이터 배포 범위는 관련 권리와 이용 조건을 별도로 검토한다.

---

## 7. 시스템 구성

```text
React + TypeScript + Vite Web UI
        │
        ├─ catalog.json
        ├─ game-data-schema validation
        ├─ Ability effect resolver
        ├─ result view model
        │
        └─ calculator package
             ├─ PMF
             ├─ DiceExpression
             ├─ roll rules
             ├─ Critical Hit effects
             └─ damage allocation
```

전투 계산은 사용자의 브라우저에서 수행한다. 별도 백엔드 서버와 외부 데이터베이스는 현재 사용하지 않는다.

---

## 8. 데이터 관리 전략

### 현재 구현

현재 샘플 카탈로그는 웹 번들에 포함된 단일 외부 JSON 파일로 제공된다. 로딩 시 스키마 검증을 통과해야 하며, 누락 ID나 잘못된 수치가 있으면 실행 전에 오류로 처리한다.

### 다음 단계

향후 전체 게임 데이터를 다룰 때는 다음 구조로 확장한다.

```text
versions.json
→ release manifest
→ common data
→ faction chunks
→ runtime validation
→ IndexedDB cache
```

목표:

- 최초 접속에서 전체 데이터를 내려받지 않는다.
- 선택한 진영에 필요한 청크만 불러온다.
- 파일 해시로 무결성을 확인한다.
- 마지막 정상 데이터를 IndexedDB에 보관한다.
- 갱신 실패 시 기존 정상 버전으로 복구한다.
- 과거 데이터 버전을 선택할 수 있도록 확장한다.

이 기능은 아직 구현되지 않았으며 현재 로드맵의 다음 단계다.

## 9. Ability와 규칙 확장 전략

게임 데이터에는 임의 JavaScript를 저장하지 않는다. 계산 가능한 규칙은 허용된 선언형 효과만 사용한다.

현재 지원 효과:

```text
hit-reroll
wound-reroll
critical-hit-threshold
sustained-hits
lethal-hits
```

향후 규칙은 계산 파이프라인의 적용 시점과 상태를 먼저 정의한 뒤 효과 타입, 검증, 계산 테스트와 UI 표시를 함께 추가한다.

후속 후보:

- Critical Wound
- Devastating Wounds
- Mortal Wounds 별도 경로
- 피해 감소와 최소 피해
- Feel No Pain
- 복수 공격 그룹 순차 처리

## 11. 배포 및 운영

현재 배포 환경은 **GitHub Pages**다.

```text
Pull Request / main push
→ GitHub Actions
→ npm ci
→ npm run check
→ Vite production build
→ GitHub Pages artifact
→ deploy-pages
```

CI와 배포 워크플로는 Node.js 24를 사용한다. 로컬 개발은 저장소 `engines` 조건인 Node.js 20.19 이상 또는 22.12 이상을 지원한다.

초기 운영 범위에서 제외하는 요소:

- 백엔드 서버
- 외부 데이터베이스
- 사용자 계정과 클라우드 동기화
- 유료 API와 유료 저장소
- 관리자용 웹 편집 화면
- 실시간 다중 사용자 기능

## 15. 기대 효과

플레이어는 복잡한 주사위와 피해 할당 계산을 직접 수행하지 않고도 무장과 대상의 전투 성능을 비교할 수 있다. 평균만이 아니라 결과별 확률과 최빈 상태를 제공하므로 공격 대상 선택과 자원 투입 판단에 활용할 수 있다.

개발 측면에서는 확률분포 합성, 상태 전이, 규칙 데이터 모델링, 런타임 검증, React UI, 테스트 자동화와 정적 웹 배포를 분리된 구조로 경험하고 확장할 수 있다.
